# Task Management Dashboard

A responsive task-management dashboard for Customer Success teams. TaskFlow gives new and experienced CSMs a shared workspace for triaging customer work, finding overdue commitments, updating status inline, and creating or editing tasks through accessible forms.

The application is a frontend demonstration backed by realistic local data. It includes responsive table/card views, debounced search, composable filters, optimistic updates, task detail and form dialogs, loading/error/empty states, keyboard shortcuts, accessible interaction patterns, and a persistent light/dark theme.

Theme selection initially follows the operating-system preference. Use the sun/moon control in the navigation to switch modes; the explicit choice is saved in `localStorage` for future visits.

## Tech Stack

> **Version note:** the project uses the React 18+ application model and APIs requested for the exercise, while the installed package is currently React `19.2.7`. The UI layer follows shadcn/ui's composable local-component approach, but the primitives in `src/components/ui/` are project-owned implementations rather than code generated by the shadcn CLI.

- **React 18+** — provides component composition, hooks, portals, and accessible interactive UI patterns; the repository currently resolves React 19.2.7.
- **TypeScript** — enforces task-domain contracts, typed component props, and safe payload transformations across the application.
- **Vite** — supplies a fast development server, React HMR, and an optimized production build with minimal configuration.
- **Zustand** — manages shared task data and asynchronous mutations with much less ceremony than Redux for this application's scope.
- **Tailwind CSS** — provides responsive, co-located utility styling and a consistent visual system without maintaining large feature-specific stylesheets.
- **shadcn/ui-style primitives** — the locally owned `Button`, `Input`, `Select`, `Textarea`, `Modal`, `Toast`, and related components provide consistent, composable UI contracts that remain easy to customize.
- **React Hook Form** — handles performant form state, dirty tracking, blur validation, and submission without a controlled state variable for every field.
- **Zod** — defines a single runtime validation schema with typed output and user-facing validation messages.
- **date-fns** — handles readable due dates, business-day defaults, date comparison, and overdue-day calculations with small composable functions.
- **Lucide React** — supplies a coherent, lightweight icon set whose SVGs compose naturally with React and Tailwind.

### Techniques Used

- **Behavior-first DOM assertions:** tests query controls by semantic role and accessible name and assert what a user can observe instead of component internals or snapshots.
- **Realistic interaction simulation:** `user-event` covers typing and clicking; `fireEvent` is reserved for lower-level form, keyboard, and backdrop events where exact event control matters.
- **Deterministic asynchronous testing:** fake timers drive the 300 ms debounce, deferred promises expose optimistic state before settlement, and `waitFor`/`findBy*` synchronize React effects and asynchronous feedback.
- **Service-boundary isolation:** `vi.spyOn` controls `taskService` success and failure responses without bypassing Zustand actions, so tests exercise the real optimistic-update and rollback workflow.
- **Canonical rollback verification:** failure tests assert both the temporary optimistic state and the subsequent `getAll()` reload, including preservation of the original mutation rejection when rollback loading also fails.
- **Accessibility-focused modal testing:** focus entry, Tab/Shift+Tab wrapping, Escape prevention, backdrop semantics, focus restoration, `aria-modal`, dialog naming, and scroll-lock cleanup are verified directly.
- **Dirty-form path coverage:** every dismissal path is routed through the same unsaved-changes guard and tested for both pristine and dirty form states.
- **Reusable typed fixtures:** `src/test/factories.ts` supplies task builders and controllable deferred promises; `src/test/setup.ts` installs jest-dom matchers, cleanup, and browser API shims.
- **Strict coverage gate:** Istanbul thresholds fail `npm run test:coverage` unless the selected modules remain at 100% statements, branches, functions, and lines.

## Assumptions

1. **Assignees are a fixed list of four team members.** User and team management are outside the project scope, so `ASSIGNEES` is a constants-driven list.
2. **Filters are AND-ed.** A high-priority filter combined with Open status shows tasks that are both high priority **and** open.
3. **Search is real-time with a 300 ms debounce.** Keystrokes update the visible input immediately while filtering waits briefly to avoid unnecessary recomputation.
4. **Tasks cannot be deleted.** Work leaves the active workflow by being marked Completed, preserving the basis for an eventual audit trail.
5. **A task is overdue when its due date is in the past and its status is not Completed.** Completed work never contributes to the overdue count.
6. **Completed and In Progress tasks can be reopened.** Changing either status back to Open is allowed, with confirmation in the detail panel.
7. **Each task has exactly one assignee.** Multi-owner tasks, teams, and watchers are not represented in the current domain model.
8. **Due dates display in the user's local timezone.** ISO strings are parsed and formatted in the browser rather than against a configurable account timezone.
9. **Option C—local JSON-style TypeScript data—is the selected data source.** See [Data Source Decision](#data-source-decision) for the rationale and migration path.
10. **Authentication and authorization are out of scope.** Every visitor can see and mutate every locally loaded task.
11. **New tasks start with Open status.** Status is assigned in the service and is intentionally absent from the creation form.
12. **New customers default to the Starter plan.** Editing preserves an existing customer's plan; customer-plan selection is not part of the task form.
13. **The browser session is the persistence boundary.** In-memory mutations survive navigation within the running app but reset after a full server restart or reload of source data.
14. **Desktop is defined as 1024 px and wider.** Desktop uses the sortable table and right-side detail panel; narrower screens use cards and a full-screen detail experience.

## Missing Requirements I Identified

| Ambiguous requirement | Decision made |
| --- | --- |
| Are filters OR-ed or AND-ed? | AND-ed all active search, priority, status, and assignee criteria so results become progressively narrower and predictable. |
| What happens to overdue tasks? | Added a red visual indicator, an overdue count in statistics, and a prominent alert banner with navigation to the task list. |
| Can completed tasks be reopened? | Yes. The inline control allows status changes, and the detail panel asks for confirmation before moving In Progress or Completed work back to Open. |
| How should timezones be handled? | Dates are parsed and displayed in the browser's local timezone; account-level timezone configuration is deferred. |
| What is the assignee list? | A documented, fixed list of four Customer Success team members from `ASSIGNEES`. |
| What is the maximum description length? | 1,000 characters, enforced by both the form input and Zod schema. |
| Can tasks be deleted? | No. The product uses a status-based workflow only, avoiding destructive actions and preserving work history. |
| What is the default sort order? | Seeded service order is preserved initially and users can sort every primary table column. A production default would be priority descending (High first), then due date ascending, implemented in the query/service layer. |

## Architecture

### Project Structure

```text
task-dashboard/
├── public/                         # Static files copied directly into the build
├── src/
│   ├── components/
│   │   ├── dashboard/              # Dashboard summaries and overdue alert
│   │   ├── features/tasks/         # Task filters, list/table/cards, detail, and form
│   │   ├── layout/                 # App shell, responsive sidebar, and main landmark
│   │   └── ui/                     # Reusable project-owned UI primitives
│   ├── constants/                  # Assignees and status/priority/filter configuration
│   ├── data/                       # Realistic local task fixtures and delay simulation
│   ├── hooks/                      # Task orchestration, filtering, and debounce hooks
│   ├── services/                   # Async data-source boundary (`taskService`)
│   ├── store/                      # Zustand task state and mutation workflow
│   ├── types/                      # Task-domain types and create/update payloads
│   ├── utils/                      # Shared helpers such as Tailwind class merging
│   ├── App.tsx                     # Application composition and transient UI state
│   ├── index.css                   # Tailwind entrypoint and minimal global defaults
│   └── main.tsx                    # React root and Strict Mode bootstrap
├── index.html                      # Vite HTML entrypoint
├── vite.config.ts                  # React and Tailwind Vite plugins
├── tsconfig*.json                  # TypeScript project configurations
└── package.json                    # Dependencies and project scripts
```

Folders are organized by responsibility. Domain types and data access do not depend on presentation components; feature components can compose shared UI primitives; hooks bridge shared state into screen-ready data; and `App.tsx` is limited to application composition and short-lived interaction state.

### Data Flow

```text
Component
   ↓ calls/reads
useTasks hook
   ↓ orchestrates
Zustand task store
   ↓ delegates I/O
taskService
   ↓ reads/writes
mockTasks local data
```

The flow is unidirectional:

1. `App.tsx` and task components consume the state and actions exposed by `useTasks()`.
2. `useTasks()` combines the Zustand store with filter/search state, derives `filteredTasks` and `overdueCount`, starts the initial fetch, and updates the document title.
3. The store owns shared asynchronous task state (`tasks`, `loading`, and `error`) and mutation behavior.
4. The store delegates persistence operations to `taskService` instead of importing mock data directly.
5. The service reads or updates `mockTasks`, simulates network latency, and returns typed promises.
6. Store updates trigger React subscriptions, derived values are recalculated, and the UI renders the new state.

This separation prevents components from knowing whether data comes from an array, REST API, GraphQL endpoint, or another adapter.

### State Management

**Why Zustand over Redux:** this application has one compact shared domain, a small action surface, and no need for reducers, action constants, middleware configuration, or normalized entity tooling. Zustand keeps asynchronous actions close to the state they update while retaining predictable subscriptions and testable service boundaries. Redux Toolkit would be appropriate if the product developed many interacting domains, complex event history, or organization-wide Redux conventions.

State is divided deliberately:

- **Zustand store:** tasks, initial loading state, load errors, and create/update/status actions shared across the application.
- **Hook-level state:** search text, debounced search, priority/status/assignee filters, active-filter count, and filtered results.
- **Local component/application state:** selected task, create/edit/detail visibility, submitting state, table sort, mobile navigation, and cancel/reopen confirmations.
- **URL state:** none in this version. Filters and selected tasks do not survive refresh and are not shareable links. A production version should consider query parameters for filters and a route such as `/tasks/:id` for task details.

`updateTask` and `updateStatus` use optimistic updates. The store immediately merges changes and stamps `updatedAt`, allowing rows, cards, details, and statistics to react without waiting for simulated I/O. If the service rejects the operation, the store refetches the canonical task list and rethrows the error so the UI can notify the user. Creation is intentionally pessimistic: a new task enters the store after the service succeeds.

### Data Source Decision

The project chooses **Option C: local JSON-style data** because the exercise focuses on interaction design, architecture, and frontend implementation rather than backend provisioning. The fixtures provide deterministic edge cases—overdue work, long titles, multiple plans, and every status/priority combination—without credentials, CORS, deployment, or API availability becoming setup blockers.

Local data is not imported by components or hooks. `taskService.ts` is an asynchronous abstraction with `getAll`, `getById`, `create`, `update`, and `updateStatus` methods. It deliberately simulates network delays and returns promises, so the UI already exercises loading and submitting behavior.

To swap in a real REST API, replace the internals of `src/services/taskService.ts` with `fetch` or an HTTP client while preserving its typed method signatures. The store, hooks, and components can remain unchanged. Production work would also add request cancellation, response validation, authentication headers, error normalization, pagination parameters, and conflict handling.

### Key Architectural Decisions

- **Service abstraction layer for an API swap:** only `taskService.ts` knows the concrete data source, keeping infrastructure concerns out of Zustand and React components.
- **Optimistic updates with rollback:** edits and status changes render immediately; failed writes refetch canonical data and surface an error toast.
- **Single `useTasks` hook as the orchestrator:** the screen receives shared state, derived data, filters, counts, and domain actions from one cohesive integration point.
- **Constants-driven configuration:** `PRIORITY_CONFIG`, `STATUS_CONFIG`, `ASSIGNEES`, and `DEFAULT_FILTERS` centralize labels and domain options rather than scattering magic strings.
- **Typed domain boundaries:** `Task`, `CreateTaskPayload`, and `UpdateTaskPayload` keep service, store, form, and component contracts aligned.
- **Responsive presentation, shared behavior:** desktop rows and mobile cards differ visually but receive the same task and callbacks, preventing duplicated business logic.

## UX Decisions

### Dual-Persona Design

The dashboard is designed for both new CSMs who need context and experienced CSMs who prioritize throughput.

- **New CSMs** can open the detail panel to read the full description, customer plan and contact information, assignment details, overdue duration, and metadata before acting. Explicit badges, section labels, confirmation before reopening, empty-state guidance, and full form labels reduce ambiguity.
- **Experienced CSMs** can scan a dense desktop table, sort columns, update status inline, search with `/`, open creation with `N`, and use card/table rows as large selection targets. They can complete routine triage without opening the detail panel.
- **Both personas** see the same status language, priority colors/icons, responsive filters, statistics, and toast feedback, so moving between quick and detailed workflows does not require relearning the product.

### Overdue Task Treatment

Overdue handling was added because due dates without escalation affordances can hide the highest-risk customer commitments. The implementation defines overdue as `dueDate < now && status !== 'completed'` and applies that rule consistently in `useTasks`, `StatsBar`, task rows/cards, and `TaskDetail`.

Users receive multiple levels of signal:

- A dashboard alert announces the current overdue count.
- The statistics bar includes a dedicated Overdue metric.
- Rows and cards pair red styling with explicit “Overdue” text rather than relying on color alone.
- Task detail shows “Overdue by X days” beneath the formatted due date.
- The alert action clears constraining filters and scrolls to the task list.

### Status Change — Two Paths

Status can be changed in two places:

1. **Inline table row or mobile card:** optimized for experienced users processing many tasks quickly.
2. **Task detail panel:** optimized for users who want customer and assignment context before committing a change.

Both paths call the same store action. The detail path adds a confirmation when changing In Progress or Completed back to Open because reopening reverses a meaningful workflow transition; forward progress remains low-friction.

### Empty States

The application distinguishes four rendering states, including three action-oriented empty/error states:

- **No tasks:** explains that the workspace is empty and offers **Create Task**.
- **No results:** references the current search and offers **Clear Filters**, preserving the distinction between an empty dataset and an over-constrained view.
- **Load error:** explains that tasks could not be loaded and offers **Try Again**.
- **Loading:** uses skeleton content and a polite status message so the interface does not appear broken during simulated latency.

Different causes require different recovery actions; a generic blank table would force the user to diagnose the system.

### Information Hierarchy

The desktop table orders columns as **Priority → Task → Status → Due Date → Assignee → Actions**:

1. Priority is first for rapid risk triage and receives a narrow visual footprint.
2. Task title and customer occupy the largest column because they answer “what work and for whom?”
3. Status follows the work identity and is editable inline.
4. Due date exposes urgency and overdue state.
5. Assignee clarifies ownership after the work and timing are understood.
6. The explicit View action sits last, following common table-action conventions.

The table preserves source order initially and supports stable, non-mutating client-side sorting for Priority, Task, Status, Due Date, and Assignee. For production, priority-descending then due-date-ascending should become the service/query default.

### Keyboard Shortcuts

- **`N`** opens the New Task form, matching the primary creation action.
- **`/`** focuses search, following conventions from productivity tools and command-oriented applications.
- **`Esc`** closes the active modal, detail panel, mobile navigation, or nested confirmation; confirmations consume Escape before their parent surface closes.

`N` and `/` are implemented in `App.tsx` with a document keydown listener. They do not fire while an input, textarea, select, or content-editable element is active, when modifier keys are held, or while another dialog is open. Escape handling lives with each modal surface so focus restoration and nested-dialog behavior remain encapsulated. The visible UI remains fully usable without knowing any shortcut; shortcuts are accelerators, not requirements.

## Performance Considerations

### Implemented

- **Debounced search (300 ms):** `useDebounce` delays filter recomputation while preserving responsive input text.
- **Optimistic updates:** edits and status changes update the interface before the asynchronous service responds, with refetch rollback on failure.
- **`React.memo` on `TaskRow`, `TaskCard`, and `StatsBar`:** reduces avoidable rerenders when parent composition changes but relevant props remain stable.
- **`useMemo` for filtered task list and overdue count:** recalculates derived collections only when tasks or filter criteria change; statistics also memoize their reduction.
- **Stable/non-mutating sort behavior:** the table sorts a copied array, modern JavaScript's stable sort retains source order for equal keys, and Zustand's source array is never mutated.
- **Stable callbacks:** key selection, modal-close, creation, edit, and status handlers use `useCallback` where referential stability benefits memoized children.

### Production Improvements

- **Server-side pagination and filtering:** the current client-side scan and render strategy should not be expected to scale comfortably beyond roughly 500 tasks.
- **React Query:** add request caching, deduplication, background refetch, stale-while-revalidate behavior, mutation lifecycle management, and clearer server/client state separation.
- **Virtualized list with `@tanstack/react-virtual` or `react-virtual`:** render only visible rows for datasets of 1,000 or more tasks.
- **Code splitting:** lazy-load `TaskForm` and `TaskDetail`, which include form validation, date utilities, and portal behavior not needed for the first static dashboard frame.
- **Bundle analysis and tree-shaking audit:** inspect the production bundle, confirm icon/date imports remain granular, and set performance budgets in CI.
- **Move sorting to the query layer:** support deterministic multi-column order, pagination-safe ordering, and locale/timezone rules on the backend.
- **Measure rather than assume:** add Web Vitals, React Profiler checks, and real device/network budgets before applying broad memoization.

## Accessibility Considerations

### Implemented

- **Semantic HTML:** the application uses `table`, `thead`, `th`, `button`, `form`, `fieldset`, `label`, `nav`, `aside`, `header`, `main`, `section`, `article`, `dl`, and `time` where appropriate.
- **Skip-style navigation link:** the TaskFlow logo and Dashboard navigation link target `#main-content`, allowing keyboard users to move directly to the main landmark. A production polish item is to add a dedicated visually hidden “Skip to main content” link as the first focusable element.
- **Keyboard-accessible interactions:** rows, cards, sort headers, filters, dropdowns, dialogs, navigation, and confirmation actions work without a pointer.
- **Focus management and traps:** the shared modal, task detail panel, reopen confirmation, and mobile sidebar constrain Tab navigation, focus the relevant first target, close on Escape, and restore prior focus.
- **ARIA labels on icon-only buttons:** close, clear-search, menu, dismiss, and similar controls expose meaningful accessible names.
- **`aria-live="polite"` toast area:** success and informational toasts are announced without interrupting the user; errors use `role="alert"`. Filter result changes are represented by the updated task region and empty-state status; a dedicated live result-count announcement remains a production enhancement.
- **`aria-sort` on table columns:** sortable headers expose `ascending`, `descending`, or `none` to assistive technology.
- **Color is never the only indicator:** priority and status colors are paired with text and icons; overdue states include explicit text; validation errors include messages and invalid-state semantics.
- **Accessible form validation:** every field has a visible label, required fields expose `aria-required`, and errors are linked through `aria-describedby`.
- **Responsive modal semantics:** task details use `role="dialog"`, reopen confirmation uses `role="alertdialog"`, and background controls become inert during nested confirmation.

### Known Limitations

- The application has not been manually tested with NVDA, JAWS, or VoiceOver.
- Color contrast targets WCAG 2.1 AA, but not every secondary or disabled treatment is guaranteed to meet AAA.
- Some mobile touch targets are 40 px; 44 px minimum targets would be safer for motor accessibility.
- The app does not yet include an automated accessibility test suite with axe-core or Playwright.
- Filtered-result counts do not currently have a dedicated `aria-live` announcement.
- The navigation-to-main link is visible branding/navigation rather than a conventional first-focus hidden skip link.
- Native select behavior and date inputs vary by browser and should receive cross-browser assistive-technology testing.

## Production Readiness Review

> **If this application were launching to production tomorrow, what concerns would you raise?**

1. **No real backend:** local in-memory data is not durable, transactional, backed up, or shared between users.
2. **No authentication or authorization:** there is no identity, tenant boundary, role model, or permission check for reading and mutating customer work.
3. **Client-side filtering will not scale:** every task is loaded, scanned, sorted, and potentially rendered in the browser.
4. **No real-time collaboration:** concurrent edits can overwrite each other because there is no version field, websocket synchronization, or conflict-resolution strategy.
5. **No audit trail:** although deletion is avoided, status and field changes do not record actor, timestamped event history, previous value, or reason.
6. **Error handling needs resilience:** production requests need bounded retries, exponential backoff, idempotency, timeout/cancellation handling, and possibly circuit-breaker behavior.
7. **No offline support:** there is no service worker, local mutation queue, synchronization state, or conflict handling for intermittent connectivity.
8. **Validation is client-side only:** the backend must independently validate payload shape, due-date policy, customer access, status transitions, and field lengths.
9. **No monitoring or error tracking:** add Sentry or equivalent error reporting, Datadog/OpenTelemetry observability, structured logs, metrics, alerting, and trace correlation.
10. **Automated coverage is intentionally focused:** the current Vitest suite fully covers filter classification, store rollback, shared modal behavior, and task-form cancellation/submission paths. Broader component integration, automated accessibility checks with axe-core, real-browser Playwright coverage, visual regression, and end-to-end workflows are still needed before reliable deployment.
11. **No pagination or data retention policy:** API contracts must define cursors, limits, archival behavior, and performance targets.
12. **No security hardening review:** CSP, dependency scanning, secret handling, rate limits, CSRF strategy, secure headers, and input/output encoding need assessment.
13. **No release pipeline:** CI should enforce formatting, linting, strict type checks, tests, bundle budgets, preview deployment, and controlled rollback.
14. **Timezone semantics are underspecified:** date-only deadlines need an organization timezone and clear inclusive/exclusive cutoff rules to avoid cross-region surprises.
